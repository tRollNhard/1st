- `scripts/test_validate_skills.py` — pytest suite (12 cases) covering every branch in `validate-skills.py`: kebab-case name regex, name/directory mismatch, missing/non-string description, `allowed-tools` list-vs-string rejection, missing-license warning, missing frontmatter, and folded-scalar parsing. Runs in CI before the validator itself so a test regression blocks the PR before `custom-skills/` is scanned.
- `custom-skills/web-video-presentation/references/AUDIO.md` "See also" section — cross-references the parent `SKILL.md`, the companion `remotion-best-practices` skill (which owns the slide-rendering side that AUDIO.md's narrations get muxed onto), and the three WCAG 2.1 success criteria already cited in the body.
- `scripts/test_skill_crawler.py` — pytest suite (15 cases) covering `skill_crawler.parse_frontmatter`. The crawler and validator have independent parsers, so the existing `test_validate_skills.py` didn't guard the actual bug-fix function. Cases include folded scalars (regression test for 8223c2f), block scalars, quoted strings with colons, YAML booleans/null, edge cases (missing/unterminated/empty frontmatter), and the no-pyyaml fallback path (via `monkeypatch`). Plus a parametrized integration test that confirms the four real folded-scalar skills (`mcp-builder`, `install-skill`, `sentinel-architect`, `web-video-presentation`) still expose their full descriptions.
- `.coveragerc` — coverage configuration scoped to the two test-targeted source files (`skill_crawler.py` and `scripts/validate-skills.py`). CI now reports pytest coverage on every PR (informational, not gating). Honest baseline: validator 61%, crawler 23% — the crawler is largely untested apart from `parse_frontmatter`.
- `scripts/conftest.py` — shared pytest configuration. Inserts the repo root onto `sys.path` so test files can `import skill_crawler` without per-file path manipulation. Replaces the previous `sys.path.insert` at the top of `test_skill_crawler.py`.
- `scripts/test_skill_crawler_scoring.py` — pytest suite (21 cases) covering `skill_crawler.score_skill`, `decide`, `extract_trigger_words`, `expand_prompt`, and the `SYNONYMS` table. Closes item #8 of the 2026-05-11 outstanding-issues triage: these functions decide which skills reach Claude on every chat request, and they were previously unguarded. Cases cover each scoring branch (exact name substring +15, name-word +4, `.ext` match +10, format-kw match +10, trigger +1, whole-word desc +0.5), the `>= 2` decide threshold (edge: score=1 must not match, score=2 must match), TOP_N=3 cap, descending sort order, empty-prompt + no-match → empty list, stop-word + 4-char filtering in trigger extraction, synonym expansion (`deck` → `pptx`), the load-bearing SYNONYMS keys (`deck`, `powerpoint`, `slides`, `spreadsheet`, `excel`, `word`, `video`, `bug`, `api`, `test`), and a parametrized integration block confirming three real user prompts (`turn this PDF into a deck`, `build a video presentation`, `post a tweet about it`) still route to their expected skills. Coverage on `skill_crawler.py` rises from 23% to 40% — the remaining gap is filesystem I/O (`load_all_skills`, archive scanning) and the interactive `main`/`run` display layer.
- `server/skill-sanitization.js` — pure module that hardens skill content before it reaches the system prompt. Exports `sanitizeSkillContent` (regex-redacts the top 5 injection idioms — "ignore previous instructions", "you are now (a different)", "new role:", "system: you", chat-template tokens like `<|im_start|>` — and caps each skill at 8KB), `escapeAttr` (strips `<>&"\n\r` and length-caps at 80 chars to neutralise name-field injection), and `wrapSkill` (renders content inside a random-fenced `<skill_${hex}>` XML envelope with `<skill_name>` + `<source>` + `<skill_content>` subtags, modelled on Anthropic's official multi-document XML pattern). No side effects, no SDK deps — safe to require from anywhere.
- `server/skill-sanitization.test.js` — node:test suite (18 cases, zero devDeps) covering each redaction pattern, the byte cap, attribute escaping, fence behaviour, name-field injection, source-path leakage, and the wrapper-escape attack (literal `</skill_content>` inside content stays inside the outer fence because the fence is unguessable per-request). New `npm test` script wires `node --test "server/**/*.test.js"`.
- `.github/workflows/node-test.yml` — CI for the Node side. Runs `node --test server/*.test.js` on PRs touching `server/**`, `package.json`, or `package-lock.json`. Complements `skill-validate.yml` (Python side). Previously the node:test suites in `server/` (sanitization, automation, skill-cache-lru, skills) had no CI coverage — they only ran when a developer remembered to invoke `npm test` locally.
- `server/skill-cache-lru.test.js` — node:test suite (6 cases) covering the `skillCache` Map eviction policy in `server/providers.js`. Closes item #9 of the 2026-05-11 outstanding-issues triage. Tests assert the LRU touch (re-inserting on hit moves the entry to the tail of insertion-ordered iteration), that omitting the touch reproduces the FIFO bug, that size-cap eviction removes the head (genuine LRU after the touch), the TTL constant (60_000ms regression guard), and an integration test that fills the cache to 250 entries while continuously "reading" a hot key — the hot key survives because the touch keeps moving it to the tail, while cold older keys get evicted. `skillCache` and `SKILL_CACHE_TTL` are now exported from `providers.js` so the test exercises the real module instance instead of a re-implementation.
- `server/skills.test.js` — node:test suite (7 cases) covering `classifyCrawlerError` (timeout vs exit-nonzero vs generic), the `CRAWLER_TIMEOUT_MS` regression guard, the `matchSkills` return-shape contract (`{ alwaysActive, matched, error }`), and an integration smoke that locks in Node's `execFile` timeout signature (`err.killed === true && err.signal === 'SIGTERM'`) so a future Node version changing that convention fails the test loudly instead of silently breaking the timeout-detection branch.

Defense in depth: (1) SSoT module — `MODELS` is `Object.freeze`d and exposes `SONNET` / `OPUS` / `HAIKU` / `DEFAULT`; one place to edit when Anthropic ships a new ID. (2) `scripts/check-model-ids.js` + `npm run check:models` — Layer-2 grep gate that scans every `git ls-files` entry outside the allowlist for bare `claude-(opus|sonnet|haiku)-N` strings; treats `renderer/index.html` as a known second source and asserts every `<option value>` matches the SSoT instead of forbidding bare IDs there. (3) `validateAgainstApi(client)` exported for future runtime cross-check against `client.models.list()` (not wired in this PR; keeps scope tight). Refactored call sites: `server/providers.js:158` (`defaultModel`), `server/automation.js:59` (`claudeComplete` Haiku call), `scripts/code-improver.js:32` (`MODEL` constant). Renderer dropdown stays hardcoded with an in-file comment pointing at `server/models.js` — populating the dropdown from `/api/models` would add an Express endpoint + preload bridge + renderer init for two strings; deferred. Industry-pattern validation: Vercel AI SDK's `customProvider` uses the same semantic-alias-to-concrete-ID shape; `Object.freeze` is the standard JS-without-TypeScript way to express an immutable constant set (MDN).
- Dev-time auto-reloader for the root Electron app swapped from `electron-reload` (`2.0.0-alpha.1`, abandoned 5 years) to `electron-reloader@^1.2.3`. Same `try/catch` + `NODE_ENV === 'development'` gating; ignore filter narrowed from `/server|node_modules/` to `[/server/]` since `electron-reloader` ignores `node_modules` and dot-files by default. The dependency was also moved from `dependencies` to `devDependencies` to match its actual lifecycle. `npm audit` clean.
- `scripts/mcp_watchdog.ps1` — `Ensure-Uv` pins the winget install to `--scope user`. Current winget default is per-user (no UAC), but the hidden-window watchdog can't answer a prompt if a future winget version flips that default. Tightens blast radius (user-site only) at the same time.
- `custom-skills/install-skill/SKILL.md` — dropped the static "Anthropic name → GitHub guess" table. Cached owner mappings rot fast (forks die, repos rename); the table guaranteed a future Claude would follow stale advice. The "always run `gh skill search` first" directive that lived at the bottom of the deleted section is now promoted into Step 2A and hardened: common-owner names are framed as search hints, never install destinations.
- `.github/workflows/skill-validate.yml` — now installs Python deps from `requirements.txt` (single source of truth) and adds a `Run validator unit tests` step before the validator. Path filters include `scripts/test_validate_skills.py` and `requirements.txt` so test-only or dep-only changes trigger CI. Subsequently extended to also run `test_skill_crawler.py` and report `pytest-cov` coverage (informational); path filters now also include `skill_crawler.py`, `scripts/test_skill_crawler.py`, and `.coveragerc`. Further extended to run `test_skill_crawler_scoring.py` alongside the others (closes #8 of the 2026-05-11 triage).
- `server/providers.js` `buildSystemPrompt` — replaced the markdown-section skill rendering (`## [Always Active] {name}\n{content}`) with the random-fenced XML envelope from `server/skill-sanitization.js`. The wrapper instruction was rewritten from "Follow their guidance" (which granted skill content authority over user intent) to an explicit data-vs-instructions framing telling Claude to treat tag content as reference data only, ignore embedded directives, and not let skill text override the user request or the base system prompt. Pure helpers extracted to the new module so they can be unit-tested without dragging in the Anthropic SDK + Composio + Python crawler chain.
- `server/providers.js` + `server/skill-sanitization.js` now sanitize and wrap Composio tool-call results before they enter chat history (closes outstanding-issues #13, filed during the independent-review pass of PR #9). Same threat shape as the skill-content vector: a malicious party can plant injection text in data a tool returns (email body, web page, search result) and that text streamed straight into history unfiltered, replaying on every subsequent message until the 2-hour session expired. Fix is symmetric with the skill defence: a new `sanitizeToolResultContent` (same INJECTION_PATTERNS, a separate `TOOL_RESULT_CHAR_CAP = 65536` since tool output is legitimately larger than skill files) plus `wrapToolResult(toolName, content, fence)` rendering output inside `<tool_result_${fence} tool="…">` envelopes. The fence design switched from per-request to **per-chatId** (stored in a new `chatFences` Map with the same `CHAT_TTL_MS` cleanup as `chatHistories`) so that past tool_result blocks in history stay coherent with the system-prompt instruction — a per-request fence would have left old wrapped blocks referencing dead fences. The system prompt now always includes a "Untrusted Tool Output" framing paragraph (separate from the skill-block framing, which is still conditional on matched skills) telling Claude to treat content inside `<tool_result_${fence}>` tags as data, never as instructions. Helper functions refactored into `sanitizeUntrustedContent(raw, cap)` with `sanitizeSkillContent` and `sanitizeToolResultContent` as thin wrappers. Test suite expanded from 38 to 45 cases (7 new tests covering the larger cap, malicious tool-name escape, fence tag-escape attack, and non-string input coercion).
- `server/skill-sanitization.js` hardened in a post-PR-#9 independent-review pass: (a) `escapeAttr` now strips a much wider Unicode set via a `STRUCTURE_BREAKING_CODES` Set (51 code points covering U+2028/U+2029 line/para separators, U+0085 NEL, U+200B-200F zero-widths and bidi marks, U+202A-202E bidi formatting including the RTL override, U+2060 word joiner, U+FEFF BOM, full C0 controls) — closes the bypass surface where a skill name like `foo # SYSTEM OVERRIDE` would have rendered as a markdown line break. Implementation switched from regex character class to numeric Set because literal U+2028 in JS source terminates regex literals at parse time. (b) `SKILL_BYTE_CAP` renamed to `SKILL_CHAR_CAP` — JS `String.length` is UTF-16 code units, not bytes; the cap is char-counted, and the docstring now points at `Buffer.byteLength` for callers who need a real byte cap. (c) `INJECTION_PATTERNS` carries a `// KNOWN GAP` block listing the documented bypasses (verb swaps, contractions, word inserts, homoglyphs, Anthropic-internal `\n\nHuman:` tokens, Llama/Gemma tokens, non-English) so a future reader can't mistake the 5 patterns for an exhaustive list. (d) Test suite expanded from 18 to 38 cases, including 10 xfail-style "KNOWN GAP" tests that assert today's bypass behaviour — a future contributor strengthening the patterns will fail those tests and know to either update them or document why. (e) `server/providers.js` `buildSystemPrompt` now carries a comment explaining the per-request fence's prompt-cache trade-off (every system prompt is unique, defeating Anthropic's `cache_control` on the system block; rotate per session if hit-rate becomes load-bearing).
- `server/providers.js` `ClaudeProvider.stream` cancellation path — switched from chunk-boundary polling of `signal?.aborted` to a one-shot `addEventListener('abort', () => stream.abort(), { once: true })` listener registered before the for-await and cleaned up in `finally`. The SDK's `MessageStream.abort()` is now the documented call surface (Anthropic helpers.md recommends it over reaching into `.controller`), and the reactive listener means abort kills the in-flight HTTP request immediately instead of waiting for the next streamed chunk. Added a pre-loop `if (signal?.aborted) return` so an already-aborted signal doesn't open an HTTP stream at all. The SDK throws `APIUserAbortError` on abort, which is already handled by the existing catch in `server/index.js`. Closes item #2 of the 2026-05-11 triage in spirit, although the original triage diagnosis was incorrect: verification against the SDK source (v0.39.0) and Anthropic's helpers.md confirmed that `stream.controller` is a documented public property and that `stream.controller.abort()` does abort in-flight network requests — the original code was not throwing `TypeError` and was not leaking billing on abort. The changes here are hygiene/responsiveness wins, not a critical-bug patch.
- `server/skill-sanitization.js` `INJECTION_PATTERNS` expanded from 5 to 8 patterns; 7 of the 10 documented KNOWN GAPs are now caught. Follow-up to the post-PR-#9 hardening pass that introduced the xfail-style KNOWN GAP tests. (1) The instruction-override pattern's verb list grew from `ignore|disregard|forget` to also include `override|bypass|skip`, and the noun side gained an optional article/possessive slot (`the|any|your|my|our|these|those|some`) — closes the verb-swap and word-insert bypasses ("override your previous instructions", "ignore THE previous instructions"). (2) The role-reassignment pattern now matches the `'re` contraction in addition to `\s+are` ("you're now a different"). (3) A new verb-plus-role pattern catches `pretend|act as|behave as|roleplay as` followed by an elevation/jailbreak role (`root|admin|sysadmin|superuser|developer|jailbroken|unrestricted|uncensored|dan|sudo`). The role list is intentionally narrow so legitimate phrases like "pretend to be patient" don't false-positive. (4) The ChatML token pattern broadened from a fixed alternation of 5 tokens to any `<|token|>` shape (length-bounded `<\|[a-zA-Z_][a-zA-Z0-9_]{0,32}\|>` to prevent catastrophic backtracking), so Llama 3 tokens (`<|begin_of_text|>`, `<|eot_id|>`), ChatML extras (`<|endoftext|>`, `<|startoftext|>`, fim_*), and future model tokens are caught by default. (5) New patterns added for Llama 2 instruction format (`[INST]|[/INST]|<<SYS>>|<</SYS>>`) and Anthropic-legacy turn markers (`\n\nHuman:` / `\n\nAssistant:` — the doubled-newline prefix is load-bearing so single-newline interview transcripts don't false-positive). Three KNOWN GAPs remain: non-English idioms, homoglyph obfuscation (Cyrillic `і` U+0456), and zero-width char insertion (`i<U+200B>gnore`); closing those requires NFKC + confusable-mapping normalization on a copy before pattern match (for the latter two) or a multilingual idiom corpus (for the first). Test suite expanded from 38 to 54 cases: the 7 closed bypasses flipped from xfail to caught-case assertions, 5 new bypass-variant cases added (`bypass any previous rules`, `skip those previous instructions`, `act as admin`, `pretend you are DAN`, `<|begin_of_text|>`), and 4 false-positive guards added (`you are now ready`, `pretend to be patient`, `bypass the cache layer`, single-newline `Human: question` transcript).

### Fixed
- `server/automation.js` `processedGuids` set capped at 5000 entries with FIFO eviction (closes item #7 of the 2026-05-11 outstanding-issues triage). Old behaviour: a long-running automation polling an RSS feed every 30 min grew the in-memory dedup set unboundedly — a slow memory leak that would only surface after weeks of uptime. New behaviour: switched the dedup container from `Set` to `Map` (insertion-order iteration is spec-guaranteed) so a new `_rememberGuid(guid)` helper can evict the oldest entry once `PROCESSED_GUID_CAP = 5000` is reached. FIFO is the correct eviction semantics here — GUIDs are only ever looked up by exact equality for dedup, no "recently-seen" notion exists. 5000 GUIDs × ~80 bytes ≈ 400 KB ceiling, orders of magnitude more than any realistic backlog. `server/automation.test.js` (new, 7 cases via `node:test`) covers initial state, idempotent insert, cap enforcement, FIFO order, and the legacy singleton-export interface so `server/index.js` keeps working.
- `server/providers.js` `skillCache` eviction was FIFO, not LRU as the inline behaviour implied. The Map was iterated in insertion order on `keys().next()` — so when the cache filled to 200 entries, the entry that got evicted was always the *oldest by insertion*, regardless of how recently it was read. A frequently-re-asked first-message (think "hello", "help", any common opener) would be evicted before a one-shot 200th-message entry that hadn't been read since. Fix is the standard Map-LRU touch: `delete(key); set(key, cached)` on every cache HIT, which moves the accessed entry to the tail of iteration order. Now `keys().next().value` is the genuine LRU head and the eviction comment matches behaviour. Closes item #9 of the 2026-05-11 outstanding-issues triage. Bonus: `setInterval` in `providers.js` got `.unref()` so the test process exits naturally instead of waiting 10 minutes for the next chat-history sweep — production use (`server/index.js` keeping the event loop alive via the Express listening socket) is unaffected.
- `server/skills.js` `matchSkills` previously collapsed every failure path — Python missing, crawler timeout, non-zero exit, JSON parse failure — into the same empty `{ alwaysActive: [], matched: [] }` shape as "no skills matched." That meant always-active skills (truthfinder, clarifying-questions) silently dropped on any failure with no log distinction from a clean no-match. Closes item #10 of the 2026-05-11 outstanding-issues triage. Fix: (a) timeout raised from 5s → 10s — `skill_crawler.py` does filesystem I/O across every SKILL.md plus archive scans, and on a cold disk the first message of a session was hitting the 5s ceiling regularly. (b) Return shape gains an explicit `error: 'timeout' | 'exit-nonzero' | 'parse' | 'no-python' | null` field — backward compatible (existing destructure of `{ alwaysActive, matched }` keeps working) but lets the caller distinguish failure from a clean no-match. (c) Timeout-detection uses `err.killed === true && err.signal === 'SIGTERM'` — the actual `child_process.execFile` timeout signature, not the `'ETIMEDOUT'` code that the Node docs imply (that's `fs`, not `child_process`). A new test locks in the convention so a future Node version changing it fails loudly. (d) Each failure mode gets a distinct log line (timeout names the dropped always-active skills explicitly, parse errors include stdout length, exit-nonzero includes the error message).
- Fabricated Anthropic model IDs across three call sites — Sonnet `claude-sonnet-4-6-20250620` and Opus `claude-opus-4-6-20250620` are nonexistent snapshots; Anthropic dropped the date suffix starting with the 4.6 generation. Call sites: `renderer/index.html:35-36` (Sonnet/Opus dropdown options — now `claude-sonnet-4-6` and `claude-opus-4-7`, label bumped "Opus 4.6"→"Opus 4.7"), `server/providers.js:98` (provider default — now `claude-sonnet-4-6`), `scripts/code-improver.js:32` (the `npm run review` script — now `claude-sonnet-4-6`). User-visible effect: selecting Sonnet or Opus from the chat dropdown 404'd and surfaced as `[Error: Request failed. Please try again.]`; `npm run review` exited with code 2. Haiku 4.5 was unaffected — its ID (`claude-haiku-4-5-20251001`) legitimately retains the date because Haiku 4.5 predates the convention change. IDs verified against `docs.anthropic.com/en/docs/about-claude/models/overview`.
- Watchdog verify window extended from ~12s to ~30s in both `cowork_watchdog.ps1` and `mcp_watchdog.ps1`. `npm run server` cold-start takes 20-25s on this box, so the previous window logged spurious `FAILED` lines while the backend was still coming up.
- `scripts/test_skill_crawler.py` assertion tightening from the PR #6 retrospective review: (a) `test_yaml_boolean_stringified` now asserts the exact pre-lower string `"True"` instead of a loose `.lower() == "true"` that would have accepted any case; (b) the four-skill integration test replaces per-skill absolute length floors (300/300/500/400 chars) with a single loose 50-char sanity floor — the real regression signal is the indicator-char assertion, and the previous thresholds would have broken on routine wording trims; (c) the indicator-char assertion also rejects empty string.
- Documentation: CHANGELOG, ARCHITECTURE, and SECURITY now list the full Express endpoint set (chat, settings, automation, `/mcp`) instead of only the three primary chat-flow endpoints — the security boundary is wider than the original doc claimed.
- `skill_crawler.py` `parse_frontmatter` — switched from line-based parsing to `yaml.safe_load` when pyyaml is available. The old parser only saw `description: >` and stored the literal `>` indicator character as the value, silently breaking keyword matching for three skills that already used folded scalars (`mcp-builder`, `install-skill`, `sentinel-architect`). All three now expose their full descriptions to matching (390-602 chars each). Falls back to the original line parser when pyyaml is missing so chat doesn't crash on a fresh clone before `setup.sh` runs. All values are stringified so downstream `.lower()`/`.strip()` calls don't choke on YAML booleans or lists.
- Post-review cleanup of today's tightening pass (independent reviewer findings): (a) `setup.sh` Python-dep install now detects whether `python` is inside a virtualenv (`sys.prefix != sys.base_prefix`) and omits `--user` when it is, since `pip install --user` is rejected under venvs and would have silently failed to install pyyaml; the failure-message recovery command also matches the actual install flag used. (b) `.github/workflows/skill-validate.yml` `push:` paths block now includes `.github/workflows/skill-validate.yml` itself — workflow-only edits pushed to main will now correctly trigger the run. (c) `test_skill_crawler.py` indicator-char regression check uses `desc.strip()` so a future fallback-parser tweak that leaves trailing whitespace (`"> "`, `">\n"`) still gets caught.

### Removed
- `next1.md` — the single QUEUED watchdog-verification step it described was empirically completed by the watchdog itself at 2026-05-08 18:00 (`DOWN - backend not responding, restarting...` → `RESTARTED - backend healthy after 14s` → sustained `OK` lines). Handoff is no longer load-bearing.

### Security
- Smilee/visualization is and remains fully offline — no network calls under any circumstance.
- `Motivate/server/publisher.js` Imgur Client-ID moved from hardcoded literal (`546c25a59c58ad7`) to `process.env.IMGUR_CLIENT_ID`. Closes item #5 of the 2026-05-11 outstanding-issues triage. Imgur has been revoking anonymous shared Client-IDs since 2023; the embedded ID was at imminent risk of failing silently in the Instagram-publish path. `Motivate/.env.example` updated with the new var and a pointer to `https://api.imgur.com/oauth2/addclient`. See `Motivate/CHANGELOG.md` Unreleased section for the in-product entry.
- `server/providers.js` skill content now goes through three layers of defence before reaching Claude (closes item #6 from the 2026-05-11 outstanding-issues triage). Vector 1 (classic injection — `Ignore all previous instructions...` inside a SKILL.md) is partially neutralised by `sanitizeSkillContent`'s pattern redaction and the 8KB cap. Vector 2 (trust elevation via the old "Follow their guidance" framing — a malicious skill needed no injection idioms to dictate behaviour, it could just include assertive text) is killed by rewriting the wrapper header to explicit data/instruction separation. Vector 3 (name-field injection via attacker-controlled YAML frontmatter — a skill named `\n\n# SYSTEM OVERRIDE\n...` would break the markdown structure of the prompt) is killed by `escapeAttr` plus the move from markdown headers to XML subtags. The XML envelope itself uses a `crypto.randomBytes(8)` fence per request so a skill author cannot pre-write a matching closing tag to escape the wrapper. Patterns and approach drawn from Anthropic's prompt-engineering docs (XML structure, mitigate-jailbreaks), OWASP LLM01 cheat sheet (data-not-instructions framing), Microsoft's Spotlighting research (delimiter randomisation), and the 13-LLM delimiter-defence study (95%+ block rate on current-gen models when wrapped + framed correctly). Ceiling for layered defence against natural-language re-framing is still ~95-98%; a Haiku 4.5 pre-screen at skill-load time (cached by file hash) would raise it further and is deferred to v2.

## [1.0.0] - 2026-04-29

Initial public-style release. Capabilities present at this tag:

### Added
- **Electron desktop shell** with chat UI (`renderer/`, no framework), `main.js` Electron main process, and `preload.js` exposing a minimal `contextBridge` between renderer and the local Express server.
- **Express backend** (`server/index.js`, port 3001) with:
  - `/api/chat` (POST) — streamed text responses (Transfer-Encoding: chunked).
  - `/api/providers` (GET) — health/status.
  - `/api/abort` (POST) — cancel in-flight requests.
  - `/api/settings/status` (GET), `/api/settings/keys` (POST) — read/write `ANTHROPIC_API_KEY` and `COMPOSIO_API_KEY` in `.env` from the UI.
